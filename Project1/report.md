# 프로젝트 1: 자동화 도구 비교 구현 (n8n vs Make)

## 워크플로우
"RSS 피드 신규 기사 감지 → 조건 분기(제목에 '삼성전자' 포함 여부) → Slack 알림 + Google Sheets 기록"

동일한 로직을 Make(클라우드 SaaS)와 n8n(셀프호스팅) 두 도구로 각각 구현했다.

---

## [Make 구현]

- **Trigger**: RSS 앱의 "Watch RSS feed items" 모듈 (폴링 기반, 모듈 자체에 마지막으로 처리한 항목 커서가 내장되어 중복 방지)
- **조건 분기**: Router 모듈 — 1st 경로("삼성전자" 포함) / Else 경로(불포함)
- **Action 1**: Slack — Send a Message (1st 경로에서 실행)
- **Action 2**: Google Sheets — Add a Row (1st 경로는 Slack 뒤에 이어서 실행, Else 경로는 단독으로 실행)

**구성도**: `screenshots/make-flow.png`

## [n8n 구현]

- **Trigger**: Schedule Trigger(15분 간격)
- **피드 조회**: RSS Read — 일반 액션 노드로 매 실행마다 피드 전체(약 50건)를 새로 가져옴
- **중복 방지**: Remove Duplicates 노드 — 기사 링크(`link`)를 키로 이전 실행에서 이미 처리한 항목을 자동으로 걸러냄. 처리 이력은 n8n DB(`processed_data` 테이블)에 저장되어 재시작 후에도 유지됨
- **정렬**: Sort by Date 노드 — `isoDate` 기준 오름차순(오래된 기사부터) 정렬. RSS Read가 최신순으로 넘겨주는 데이터를 뒤집어, 뒤이어 Google Sheets에 기록될 때 최신 기사가 맨 아래에 쌓이도록 맞춤
- **조건 분기**: IF 노드 — `title`에 `삼성전자` 또는 `삼전` 포함 여부
- **Action 1**: Slack — Send a Message (조건이 True일 때만 실행)
- **Action 2**: Google Sheets 기록 — HTTP Request 노드로 Google Sheets API(`values:append`)를 직접 호출. True/False 양쪽 경로 모두에서 실행되며, `분기결과` 컬럼 값만 "알림 발송" / "스킵됨"으로 다르게 기록

> 처음에는 n8n의 전용 폴링 트리거 노드인 "RSS Feed Trigger"(이전에 처리한 항목을 자동으로 건너뛰는 노드)를 썼으나, 이 노드는 **에디터에서 수동으로 "Execute step"을 누르면 중복 방지 커서를 갱신하지 않고 무조건 최신 기사 1건을 그대로 흘려보내는 설계**(소스 코드 확인: `getMode() === 'manual'`일 때 분기 처리)를 갖고 있어, 테스트 중 수동 실행을 반복하다 스프레드시트에 중복 행이 쌓이는 문제를 겪었다. 이를 Schedule Trigger + RSS Read(일반 액션) + Remove Duplicates(수동/자동 구분 없이 항상 커서 갱신) 조합으로 교체해 최종 구조를 완성했다.

**구성도 및 실행 트레이스**: `screenshots/n8n-flow.png`

## 구현 과정 요약

두 도구 모두 트리거 → 조건 분기 → 액션 2개(Slack, Sheets)라는 동일한 구조로 구현했다. 과정에서 겪은 실질적인 차이는 다음과 같다.

1. **Google 계정 연동 방식이 크게 다르다.** Make는 클라우드 SaaS라 자체 OAuth 앱이 내장돼 있어 "Sign in with Google" 버튼 한 번으로 연동이 끝났다. 반면 n8n은 셀프호스팅이기 때문에 Google Cloud Console에서 별도의 OAuth 클라이언트를 직접 등록해야 했다(API 활성화, OAuth 동의 화면 설정, 테스트 사용자 등록, 리디렉션 URI 입력 등 약 10분 소요). 이는 셀프호스팅 도구의 전형적인 트레이드오프다.

2. **RSS 트리거/액션 모듈 구분은 두 도구가 동일한 함정을 갖고 있었지만, 결론은 달랐다.** Make는 "Retrieve RSS feed items"(일반 액션)와 "Watch RSS feed items"(트리거, 중복 방지 내장)가 별도 모듈로 존재하고, n8n도 "RSS Feed Read"(일반 액션)와 "RSS Feed Trigger"(폴링 트리거, 중복 방지 내장)가 별도 노드로 존재한다. 처음엔 두 도구 모두 트리거 전용 모듈을 선택해 구성했다. Make의 Watch 트리거는 이후에도 안정적으로 동작했지만, n8n의 RSS Feed Trigger는 실제로 사용해보니 **"에디터에서 수동 실행(Execute step)하면 중복 방지 커서를 갱신하지 않고 최신 기사 1건을 무조건 흘려보내는" 설계상 허점**이 있어, 개발 중 테스트 삼아 몇 번 수동 실행했을 뿐인데 시트에 중복 행이 쌓이는 사고로 이어졌다. 결국 n8n 쪽은 Schedule Trigger + RSS Read(일반 액션) + Remove Duplicates(수동/자동 구분 없이 항상 이력을 갱신하는 노드)로 재구성해 문제를 해결했다. 즉 "트리거 전용 모듈이 곧 안전하다"는 가정이 n8n에서는 깨졌고, 오히려 액션 노드 + 명시적 중복 제거 노드 조합이 테스트 내구성 면에서 더 안전했다.
3. **중복 제거 방식이 바뀌면서 순서(정렬) 문제가 새로 생겼다.** Make의 Watch 트리거는 폴링마다 보통 신규 항목만 소량으로 흘려보내 순서 문제가 거의 발생하지 않는다. 반면 n8n의 최종 구조는 매번 피드 전체(약 50건)를 가져온 뒤 Remove Duplicates로 신규 항목만 걸러내는 방식이라, 한 번에 여러 기사가 몰릴 수 있다. 이때 RSS Read가 최신순으로 데이터를 주기 때문에 그대로 시트에 기록하면 최신 기사가 위, 오래된 기사가 아래로 뒤집혀 쌓이는 문제가 생겼다 — Sort by Date 노드를 추가해 오름차순으로 뒤집은 뒤 기록하도록 해결했다.

4. **실제 구현 방식 자체가 달랐다.** Make는 웹 화면에서 **직접 마우스로 모듈을 클릭·드래그하며 수동으로 구성**했다. 반면 n8n은 **Claude Code(AI 코딩 어시스턴트)가 n8n의 REST API를 통해 워크플로우 JSON을 직접 생성·수정**하는 방식으로 구현했다 — 캔버스를 손으로 클릭한 게 아니라, `GET/POST/PUT /api/v1/workflows` 같은 API 호출로 노드와 연결 관계를 코드로 정의해서 밀어넣은 것이다. 이게 가능했던 이유는 n8n이 워크플로우 구조 자체를 CRUD할 수 있는 REST API를 완전히 공개하고 있기 때문이다. Make는 시나리오 구조 자체를 API로 편집하는 기능이 없고, 대신 MCP(Model Context Protocol) 서버를 통해 "이미 만들어진 시나리오를 AI가 도구처럼 실행 요청"하는 기능만 제공한다 — 즉 Make에서는 AI가 시나리오를 "만드는" 것과 "실행을 요청하는" 것이 완전히 별개의 기능이고, 이번 과제 수준에서는 후자(실행 요청)만 가능했다. 이 차이 때문에 두 워크플로우는 겉보기엔 구조가 같아도 "누가/어떻게 만들었는가"가 근본적으로 달랐다.

5. **구글 시트 연결 방식(파일 선택 UI vs 문자열 직접 입력).** Make는 사람이 화면에서 직접 구성했기 때문에, 구글 계정 연동 후 **내 드라이브의 파일 목록이 자동으로 불러와져서 그중 스프레드시트를 검색·선택**하면 끝이었다. 반면 n8n은 API로 워크플로우를 구성하다 보니, n8n에도 원래 파일 선택기가 딸린 네이티브 Google Sheets 노드가 있음에도 이를 API 파라미터로 정확히 재현하는 대신 범용 HTTP Request 노드로 Google Sheets REST API를 직접 호출하는 방식을 택했다. 그 결과 스프레드시트 ID와 시트(탭) 이름을 URL 문자열에 직접 입력해야 했다(예: `.../spreadsheets/{ID}/values/n8n!A1:append`) — 시트 탭 이름을 바꿀 때마다 URL 문자열을 수동으로 찾아 고쳐야 했고, 실제로 이 과정에서 오타/불일치로 실행 오류가 난 적도 있었다. 이건 n8n 자체의 한계라기보다 "API로 자동 구성"이라는 이번 구현 방법이 낳은 차이에 가깝다 — 사람이 n8n 화면에서 직접 만들었다면 Make와 마찬가지로 파일 선택 UI를 썼을 것이다.

## 비교 항목

| 항목 | Make | n8n |
|---|---|---|
| UI/UX | 원형 모듈 기반, 직관적이고 시각적으로 깔끔함 | 노드 기반 캔버스, 자유도가 높지만 초기 학습 필요 |
| 설정 난이도 | 회원가입만 하면 즉시 사용 가능, Google 연동 원클릭 | 셀프호스팅 시 서버/인프라 지식 필요, Google 연동은 OAuth 앱 직접 등록 필요 |
| 연동 서비스 범위 | 3,000개 이상 앱 연동 프리셋(2026년 기준), 커스텀은 HTTP 모듈로 우회 | 공식 400개 이상 노드(커뮤니티 노드 포함 시 더 많음), 오픈소스라 커스텀 노드 제작 가능 |
| 무료 플랜 범위 | 월 1,000 크레딧까지 무료(2025년 8월 "오퍼레이션"에서 "크레딧"으로 명칭 변경), **활성 시나리오 2개 제한 + 최소 폴링 간격 15분** — 본 과제에서 두 도구 모두 폴링 주기를 15분으로 맞춘 이유 중 하나 | 완전 무료(셀프호스팅 기준 실행 횟수·시나리오 개수 제한 없음, 서버 비용은 별도 부담) |
| 실행 로그 확인 방식 | 실행 히스토리에서 라우터 경로별 통과 번들 수를 시각적으로 표시, 클릭 시 입출력 확인 | Executions 탭에서 노드별 입출력 데이터를 JSON으로 상세 확인 |
| 스케줄 설정 방식 | 모듈이 아니라 트리거 모듈 근처의 시계 아이콘(시나리오 전체 설정)에서 별도로 지정 — 처음엔 위치를 찾기 어려웠음 | 워크플로우 안에 "Schedule Trigger" 또는 폴링 트리거 노드 자체의 간격 설정 필드로 지정 |
| 자동화 확장성(API/AI 연동) | MCP 서버 지원 — AI 에이전트가 완성된 시나리오를 도구로 호출 가능 (구조 편집은 불가) | REST API 완전 공개 — 워크플로우를 코드로 생성·수정 가능 |
| 트리거 노드의 신뢰성 | Watch 트리거가 수동/자동 구분 없이 안정적으로 동작, 별도 보정 불필요 | 전용 폴링 트리거(RSS Feed Trigger)가 "수동 실행 시 중복 방지 커서를 갱신하지 않는" 설계상 허점이 있어, 최종적으로 액션 노드(RSS Read)+Remove Duplicates+Sort 조합으로 대체 |

## 장단점 정리

**Make**
- 장점: 설정이 빠르고 직관적, OAuth 연동이 원클릭, 실행 로그 시각화가 우수, 비개발자도 접근하기 쉬움
- 단점: 무료 플랜은 월 1,000 크레딧 + 활성 시나리오 2개로 제한적(실사용에는 부족), 시나리오 구조를 코드/버전관리 시스템으로 관리하기 어려움(클라우드 UI에 종속)

**n8n**
- 장점: 셀프호스팅 시 완전 무료(Sustainable Use License — 개인/내부 업무용은 실행 횟수·시나리오 개수 제한 없이 무료, 단 n8n을 호스팅해 제3자에게 유료로 재판매하는 것은 라이선스상 금지), REST API로 워크플로우를 프로그래밍 가능, 데이터를 직접 소유·관리
- 단점: 서버 운영 부담(프로세스 관리, 버전 호환성), Google 등 OAuth 연동 초기 설정이 복잡, 문제 해결에 개발 지식이 필요

## 어떤 상황에 적합한가

- **Make가 적합한 경우**: 빠르게 프로토타입을 만들어야 하는 경우, 비개발자가 직접 유지보수해야 하는 경우, 소규모 자동화(월 1,000 크레딧·활성 시나리오 2개 이내)로 충분한 경우
- **n8n이 적합한 경우**: 이미 서버 인프라가 있거나, 실행량이 많아 무료 크레딧 한도(월 1,000)를 넘기는 경우, 시나리오를 3개 이상 동시에 활성화해야 하는 경우, 데이터를 외부 클라우드에 맡기고 싶지 않은 경우, 개발 지식이 있어 워크플로우를 코드처럼 버전관리하고 싶은 경우

### 분기 실행 증거 (both branches)

- **Make**: 위 캡처에서 Slack 발송 6건, Sheets 매칭 기록 9건, Sheets 스킵 기록 5건이 각각 누적되어 있어, 1st(포함)/Else(불포함) 두 분기 모두 반복적으로 실행되었음을 확인할 수 있다.
- **n8n**: 위 캡처는 한 번의 실행에서 RSS Read가 가져온 50건 중 Remove Duplicates가 신규 1건만 남기고, Sort by Date를 거쳐 IF에서 False 분기(Sheets Log - Skipped)로 흐른 사례다. True 분기(Slack + Sheets 매칭)는 이전 테스트 중 "삼전" 키워드가 실제 기사 제목(`"[경제읽기] 삼전 호실적에도 전망 엇갈려...'반도체 고점론' 부각"`)에 매칭되어 Slack 알림이 정상 발송된 적이 있으나, 해당 화면은 이번 최종본 캡처에서는 포함하지 않았다. IF 노드의 두 출력(true/false)이 각각 Slack+Sheets매칭, Sheets스킵으로 연결된 구조 자체는 워크플로우 구성도에서 확인 가능하다.

## 결론: 프로젝트 2 도구 선택

두 도구를 실제로 구현·비교해본 결과를 바탕으로, 자유 주제 자동화(Project 2, 오마이보카 사용내역 알림)는 **n8n으로 구현하기로 결정**했다. 이유는 다음과 같다.

1. **인프라 재사용**: Project 1에서 이미 n8n 셀프호스팅 인프라와 Google Sheets/Slack 연동을 구축해둔 상태라, 새 프로젝트에서 계정 연동을 처음부터 다시 할 필요가 없었다.
2. **커스텀 로직 자유도**: Project 2는 "같은 사용자 기준 2시간 이내 중복 알림 억제"라는 상태 기반 판단 로직이 필요한데, n8n의 Code 노드(자바스크립트 실행)를 쓰면 이런 로직을 자유롭게 짤 수 있다. Make에도 유사한 기능(사용자 정의 함수, Data Store)이 있지만, 무료 플랜 제약(활성 시나리오 2개, 크레딧 한도)과 웹 UI 기반 구성이라는 특성상 반복적인 로직 수정·재테스트가 n8n보다 느렸다.
3. **API 기반 개발·테스트 workflow**: Project 1 과정에서 확인했듯 n8n은 REST API가 완전히 공개돼 있어, 워크플로우를 스크립트로 정의하고 `curl`로 즉시 트리거해 반복 테스트하는 것이 가능했다. Make는 시나리오 구조를 API로 편집할 수 없어 이런 빠른 반복 개발에는 적합하지 않았다.

즉 Project 1의 비교 항목 중 "REST API 완전 공개"와 "자동화 확장성" 항목이 실제 도구 선택의 결정적 근거가 되었다. 상세 구현 내용은 `Project2/README.md` 참고.
