# 프로젝트 1: 자동화 도구 비교 구현 (n8n vs Make)

## 워크플로우
"RSS 피드 신규 기사 감지 → 조건 분기(제목에 '삼성전자' 포함 여부) → Slack 알림 + Google Sheets 기록"

동일한 로직을 n8n(셀프호스팅)과 Make(클라우드 SaaS) 두 도구로 각각 구현했다.

---

## [n8n 구현]

- **Trigger**: RSS Feed Trigger (전용 폴링 트리거 노드, 15분 간격, 이전에 처리한 항목은 자동으로 건너뜀)
- **조건 분기**: IF 노드 — `title`에 `삼성전자` 포함 여부
- **Action 1**: Slack — Send a Message (조건이 True일 때만 실행)
- **Action 2**: Google Sheets 기록 — HTTP Request 노드로 Google Sheets API(`values:append`)를 직접 호출. True/False 양쪽 경로 모두에서 실행되며, `분기결과` 컬럼 값만 "알림 발송" / "스킵됨"으로 다르게 기록

**구성도**: `screenshots/n8n-flow.png`

## [Make 구현]

- **Trigger**: RSS 앱의 "Watch RSS feed items" 모듈 (폴링 기반, 모듈 자체에 마지막으로 처리한 항목 커서가 내장되어 중복 방지)
- **조건 분기**: Router 모듈 — 1st 경로("삼성전자" 포함) / Else 경로(불포함)
- **Action 1**: Slack — Send a Message (1st 경로에서 실행)
- **Action 2**: Google Sheets — Add a Row (1st 경로는 Slack 뒤에 이어서 실행, Else 경로는 단독으로 실행)

**구성도**: `screenshots/make-flow.png`

## 구현 과정 요약

두 도구 모두 트리거 → 조건 분기 → 액션 2개(Slack, Sheets)라는 동일한 구조로 구현했다. 과정에서 겪은 실질적인 차이는 다음과 같다.

1. **Google 계정 연동 방식이 크게 다르다.** Make는 클라우드 SaaS라 자체 OAuth 앱이 내장돼 있어 "Sign in with Google" 버튼 한 번으로 연동이 끝났다. 반면 n8n은 셀프호스팅이기 때문에 Google Cloud Console에서 별도의 OAuth 클라이언트를 직접 등록해야 했다(API 활성화, OAuth 동의 화면 설정, 테스트 사용자 등록, 리디렉션 URI 입력 등 약 10분 소요). 이는 셀프호스팅 도구의 전형적인 트레이드오프다.

2. **RSS 트리거/액션 모듈 구분은 두 도구가 동일한 함정을 갖고 있다.** n8n은 "RSS Feed Read"(일반 액션)와 "RSS Feed Trigger"(폴링 트리거, 중복 방지 내장)가 별도 노드로 존재한다. Make도 마찬가지로 "Retrieve RSS feed items"(일반 액션)와 "Watch RSS feed items"(트리거, 중복 방지 내장)가 별도 모듈로 존재한다. 처음에 n8n에서 스케줄러 + 일반 액션 노드 조합으로 잘못 구성했다가 매 실행마다 같은 기사를 중복 처리하는 문제를 겪었는데, 이는 도구의 차이가 아니라 "어떤 모듈을 선택하느냐"의 문제였다 — 두 도구 모두 트리거 전용 모듈(Watch/Trigger)을 정확히 골라야 중복 방지가 되며, 이름이 비슷한 액션 모듈과 헷갈리기 쉽다는 공통적인 주의점으로 봐야 한다.

3. **API 개방성 차이.** n8n은 REST API가 완전히 공개돼 있어 워크플로우 구조(노드, 연결, 파라미터)를 코드로 직접 생성·수정할 수 있었다(본 과제에서도 n8n 워크플로우 대부분을 API 스크립트로 구성했다). Make는 시나리오 구조 자체를 API로 편집하는 기능이 없고, 대신 MCP(Model Context Protocol) 서버를 통해 "완성된 시나리오를 AI가 도구처럼 호출"하는 기능만 제공한다 — 구조를 만드는 것과 실행을 트리거하는 것은 다른 문제였다.

## 비교 항목

| 항목 | n8n | Make |
|---|---|---|
| UI/UX | 노드 기반 캔버스, 자유도가 높지만 초기 학습 필요 | 원형 모듈 기반, 직관적이고 시각적으로 깔끔함 |
| 설정 난이도 | 셀프호스팅 시 서버/인프라 지식 필요, Google 연동은 OAuth 앱 직접 등록 필요 | 회원가입만 하면 즉시 사용 가능, Google 연동 원클릭 |
| 연동 서비스 범위 | 커뮤니티 포함 400개 이상 노드, 오픈소스라 커스텀 노드 제작 가능 | 2,000개 이상 앱 연동 프리셋, 커스텀은 HTTP 모듈로 우회 |
| 무료 플랜 범위 | 완전 무료(셀프호스팅 기준 실행 횟수 제한 없음, 서버 비용은 별도 부담) | 월 1,000 오퍼레이션까지 무료, 초과 시 유료 전환 |
| 실행 로그 확인 방식 | Executions 탭에서 노드별 입출력 데이터를 JSON으로 상세 확인 | 실행 히스토리에서 라우터 경로별 통과 번들 수를 시각적으로 표시, 클릭 시 입출력 확인 |
| 스케줄 설정 방식 | 워크플로우 안에 "Schedule Trigger" 또는 폴링 트리거 노드 자체의 간격 설정 필드로 지정 | 모듈이 아니라 시나리오 전체의 설정(좌하단 시계 아이콘)에서 별도로 지정 — 처음엔 위치를 찾기 어려웠음 |
| 자동화 확장성(API/AI 연동) | REST API 완전 공개 — 워크플로우를 코드로 생성·수정 가능 | MCP 서버 지원 — AI 에이전트가 완성된 시나리오를 도구로 호출 가능 (구조 편집은 불가) |

---

## 장단점 정리

**n8n**
- 장점: 완전 무료(셀프호스팅), REST API로 워크플로우를 프로그래밍 가능, 데이터를 직접 소유·관리, 실행 횟수 제한 없음
- 단점: 서버 운영 부담(프로세스 관리, 버전 호환성), Google 등 OAuth 연동 초기 설정이 복잡, 문제 해결에 개발 지식이 필요

**Make**
- 장점: 설정이 빠르고 직관적, OAuth 연동이 원클릭, 실행 로그 시각화가 우수, 비개발자도 접근하기 쉬움
- 단점: 무료 플랜 오퍼레이션 수 제한, 시나리오 구조를 코드/버전관리 시스템으로 관리하기 어려움(클라우드 UI에 종속)

---

## 어떤 상황에 적합한가

- **n8n이 적합한 경우**: 이미 서버 인프라가 있거나, 실행량이 많아 무료 오퍼레이션 한도를 넘기는 경우, 데이터를 외부 클라우드에 맡기고 싶지 않은 경우, 개발 지식이 있어 워크플로우를 코드처럼 버전관리하고 싶은 경우
- **Make가 적합한 경우**: 빠르게 프로토타입을 만들어야 하는 경우, 비개발자가 직접 유지보수해야 하는 경우, 소규모 자동화(월 1,000회 이내)로 충분한 경우
